## Problem Statement
Building a secure, open-source, and self-hosted "Discord/Mumble" replacement for large groups (up to 2000 users). Current solutions like Discord have privacy concerns, and Mumble lacks a modern zero-install browser experience.

## What Makes This Cool
The "Whoa" factor is 2000-person scale in a single browser tab with ultra-low latency. It provides a "command center" feel for gamers and a secure, audited environment for child advocacy conferencing.

## Constraints
- Must be zero-install (browser-based).
- Must be self-hosted and open-source.
- Must support 2000 concurrent users (requires UI virtualization and SFU horizontal scaling).
- Must bridge text chat to Zulip for history/audit.

## Implementation Details
### UI Performance at Scale: "Stage vs. Floor"
To handle 2,000 participants efficiently, the UI adopts a **Stage vs. Floor** architecture:
- **The Stage (Pinned):** Active speakers (max 10) are pinned to the top of the grid with large participant cards and prominent nameplates.
- **The Floor (Virtualized):** All other participants are rendered in a compact, virtualized list below the stage. DOM recycling is used to ensure < 10% main-thread blockage.
- **State Batching:** UI updates are batched to prevent re-renders on every signaling packet.

### Interaction States & Feedback
- **Global Hot State:** When the user's microphone is transmitting (PTT active), the entire viewport edge pulses with a **Red inset glow** (#d32f2f). This provides peripheral confirmation of "ON AIR" status.
- **Mic Status Lifecycle:** The PTT button in the footer follows a 3-stage lifecycle:
  1. **Checking:** Scanning for browser permissions.
  2. **Ready:** PTT available (Default).
  3. **Blocked:** Red "Mic Access Denied" alert with instructions to re-enable in browser settings.
- **Broadcast Lock:** In Broadcast Mode, the PTT button for Members changes to a "Listen Only" state with a padlock icon and explanatory tooltip.

### Roles & Permissions (Hierarchical)
- **Admin (Root):** Full system control. Can create/delete rooms, view audit logs, manage all recordings.
- **Moderator (Global Mod):** Can manage ANY room: switch modes, mute/kick participants, start/stop recordings, and manage hand-raise queues system-wide.
- **Operator (Room-Specific Mod):** Same permissions as Moderator, but **restricted to their assigned room/channel**.
- **Member:** Can listen (all modes) and speak (Discussion mode). Can "Hand Raise" in Broadcast mode (timestamped).

### UI Theme: "NextGen Stealth"
- **Default:** Dark Theme (#121212 background, #e0e0e0 text).
- **Adaptive:** UI automatically switches between Light and Dark themes based on the user's OS/System preference (`prefers-color-scheme`).
- **Customizable:** A manual toggle is provided in the header for users who want to override the system setting.

### Responsive Design: "Stage-First"
- **Desktop:** Sidebar for Room List, Main area split into Stage (Top) and Floor (Grid).
- **Mobile:** 
  - **The Stage:** Occupies the primary viewport.
  - **The Floor:** Hidden within a draggable **Bottom Sheet**. Users swipe up to view the full participant directory.
  - **PTT:** The PTT button remains pinned to the bottom of the viewport at all times, independent of the floor's scroll position.

### PTT Mechanism
To ensure "instant" latency, PTT will use **Audio Track Muting** (`track.mute()` / `track.unmute()`) rather than hardware-level toggles (`setMicrophoneEnabled`). This avoids the delay of re-initializing the audio hardware on every keypress.

## Premises
1. Browser-based = zero-install is the killer feature vs. Mumble.
2. Self-hosting + Open Source is the primary motivator over Discord.
3. 2000 users is a strict requirement for a single room/server environment.
4. Zulip bridge is for history/audit, not real-time text sync. (STT is Roadmap, not MVP).

## Approaches Considered
### Approach A: The Minimalist (RECOMMENDED)
- **Summary:** Flask + LiveKit + Vanilla JS. Full control, direct implementation of the existing plan.
- **Effort:** M (Human: ~3 days / CC: ~15 min)
- **Risk:** Med (Scaling to 2000 users is the primary technical challenge).
- **Pros:** Maximum control over latency, zero external UI dependencies.
- **Cons:** More manual UI work (virtualized lists, state management).

### Approach B: The Integrator
- **Summary:** Leverage jam.systems (OSS LiveKit frontend).
- **Effort:** S (Human: ~2 days / CC: ~10 min)
- **Pros:** Polished UI out of the box.
- **Cons:** Adapting a complex React frontend to the Flask/Mongo backend and adding virtualization/PTT-specific hooks may introduce more friction than building from scratch.

## Recommended Approach
**Approach A: The Minimalist.** Stick to the current implementation plan to ensure the "PTT Latency Loop" is optimized via Audio Track Muting and the UI remains lightweight via manual virtualization.
